## Why

本地同时开发多个服务时，开发者往往需要为每个服务单独打开终端，或使用 `nohup` 等方式后台运行，导致启动、查看状态、追踪日志、停止进程和回到服务目录都不够顺手。

`grepom` 已经负责多仓库的本地编排与定位，新增服务进程管理能力可以把“仓库在哪里”和“服务是否在跑”放到同一个工具中，减少开发时在终端、日志文件和进程列表之间来回切换的成本。

## What Changes

- 新增 `grepom svc` 子命令，并提供 `service` 别名，用于启动、查看、停止、清理和定位本地开发服务。
- 支持从命令行直接启动服务，也支持从 `.grepom.yml` 的服务配置中读取服务目录和启动命令。
- 服务默认后台运行，记录服务名、PID、进程组、工作目录、启动命令、日志路径、启动时间和最近状态。
- `list`/`status` 以表格形式展示服务名、状态、PID、路径、命令和日志路径，并在展示时检查真实进程状态。
- 支持查看日志，包括普通查看、类似 `tail -n`、类似 `tail -f` 的持续跟随，以及用编辑器打开日志文件。
- 支持普通 kill 和 `kill -9`，优先停止服务的整个进程组，避免子进程残留。
- 支持清理已经退出或失效的服务记录，并保留或按显式参数删除日志。
- 支持输出服务目录，配合现有 `gcd`/shell alias 使用，也可提供服务专用 shell helper。
- 新增 TUI 管理界面，用于在终端中浏览服务列表、查看状态、查看日志、停止服务、清理失效记录并跳转/复制服务路径。
- 更新 README 和 README_en 中的命令说明、配置示例和使用示例。

## Capabilities

### New Capabilities
- `service-process-management`: 管理本地开发服务的启动、状态、日志、停止、清理、目录定位和 TUI 操作。

### Modified Capabilities
- 无。

## Impact

- 影响 `cmd/`：新增 `svc`/`service` Cobra 子命令及子命令实现。
- 影响 `config/`：扩展配置结构，支持可选 `services` 字段，并保持现有配置兼容。
- 可能新增内部包，例如 `service` 或 `svc`，用于运行态存储、进程控制、状态探测、日志读取和 TUI 模型。
- 运行态数据写入用户状态目录或按配置隔离的状态目录，不写入 `.grepom.yml`。
- 需要处理 macOS/Linux 上进程组、信号、PID 存活检测和日志跟随行为；Windows 可先提供降级行为或明确不支持完整进程组控制。
- 可能引入 TUI 依赖，优先选择 Go 生态中维护良好且与现有 CLI 兼容的库。
